Analysis Report for Finding Deer Habitat in Nova Scotia (ArcGIS Pro)

Introduction

This study looks at a section of Caledonia, located in Nova Scotia, Canada, to analyze the most suitable location to create a protected area for the benefit of local deer species (family Cervidae).

Like most animals, deer require very specific habitats in order to thrive. To limit the scope of this project, three primary requirements were considered when designating a protected area: land use (prioritizing fields and forests), proximity to roads, and riparian zones. A combination of these qualities were used to create the final suitability raster used in analysis.

The final result was a 24 square kilometre corridor that could potentially host local deer species to both ensure their safety and decrease their presence in urban landscapes. Further analysis would be required before designating this land a protected area.

Methodology

The first step was to collect the neccessary data for analysis: roads, land use, and riparian zones. The study area is located in Nova Scotia, where data for road networks and riparian zones can publicly be found on GeoNOVA's website. Granular land use data, however, has not been publicized for the province of Nova Scotia. For this reason, data was extracted by doing an image classification on publicized satellite data of the area.

Once the image classification is complete, a suitability raster can be created. For this, each of the three criteria need to be given a suitable buffer distance and assigned a weight. These specifications were outlined in the project details. The model used create the suitability raster can be seen in figure 1.

It starts by creating a 200 metre buffer around each of the road, riparian zone and field datasets to properly consider the real-world impacts of each of these criteria. A mean is created from these zones, weighting each zone appropriately. From there, results are hexagonalized to improve readability, and inadequate zones are removed from the dataset. Figure 2 shows the final result.

Many potential habitats were found, but zones 20 and 29 were selected due to their size (shown in dark green). That said, these habitats are still too small to be valuable to the local deer population: a corridor must be made to connect them. This can easily be done by running cost distance analyses from each habitat, then using them to create a corridor, as shown in figure 3.

From here, all that's left is to choose a cutoff for the corridor to be created.

Results

Figure 4 shows the final corridor. With a size of 24 square kilometres, it is large enough and fits enough of the aformentioned criteria to be a suitable site to create a protected area for the deer population. Further research would need to be done before finalizing these works and starting to push for the change to be implemented, such as checking local bylaws, comparing against other criteria, and doing a more in-depth land-use classification.

LiDAR Calibration, Classification, and Building Vectorization (Microstation)

This project involved calibrating and classifying LiDAR using MicroStation, focusing on improving point cloud accuracy and producing a clean, classified dataset suitable for analysis. The workflow began with importing raw LiDAR points and compartmentalizing them into 1 kilometre grid squares to improve data load times.

Calibration was performed to correct for aircraft orientation effects, specifically adjusting for pitch and yaw to improve vertical and horizontal alignment of the point cloud. This step ensured that elevation values and feature positioning were as accurate as possible before classification. Careful review of cross-sections and elevation profiles helped verify that systematic distortions were minimized.

To streamline classification, I developed a macro that automatically categorized points into standard classes: noise, ground, low vegetation, medium vegetation, high vegetation, and buildings. Automating this process increased efficiency and consistency across tiles. After running the macro, I conducted a detailed manual review to correct misclassified areas: particularly along buildings, bridges, and complex terrain where automated routines can produce erroneous results.

With the data prepared, all that was left to do was vectorize buildings. Though the automatic vectorization tool does well, like when classifying, a manual check of all 150+ buildings was required, with some editing necessary. The Geospatial Population Statistics Tool is an add-on to QGIS that takes data on a selected set of points, categorizes them as being northeastern, southeastern, northwestern, or southwestern, then outputs the following information:


  • Number of points allocated to each quadrant
  • Total population of each quadrant

  • The place(s) with the lowest population

This output is saved as a text file to a user-specified location.

PL/SQL Function To Convert Between Any Units of Measure


This function accepts a value and can convert it to almost any unit of measure, whether it be radians to degrees or yards to nautical miles. The code draws from the units-of-measure table in order to provide accurate results. It also generates custom error messages if you attempt to convert between incompatible unit types, such as radians to metres. Functions such as this can streamline workflows in industry by simplifying calculations that may otherwise take up valuable time.


CREATE OR REPLACE FUNCTION COGS_CONVERT_UNIT (VALUE NUMBER, STARTING_UNIT SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE, CONVERT_TO_UNIT SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE) RETURN NUMBER IS
    START_FACT_B SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE;
    START_FACT_C SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE;

    END_FACT_B SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE;
    END_FACT_C SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_NAME%TYPE;

    START_TYPE SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_TYPE%TYPE;
    END_TYPE SDO_UNITS_OF_MEASURE.UNIT_OF_MEAS_TYPE%TYPE;

    METRES NUMBER;
BEGIN

    IF STARTING_UNIT = CONVERT_TO_UNIT THEN
        RETURN VALUE;
    END IF;

    select FACTOR_B, FACTOR_C, UNIT_OF_MEAS_TYPE
    into START_FACT_B, START_FACT_C, START_TYPE
    from SDO_UNITS_OF_MEASURE
    where UPPER(UNIT_OF_MEAS_NAME) = UPPER(STARTING_UNIT);

    select FACTOR_B, FACTOR_C, UNIT_OF_MEAS_TYPE
    into END_FACT_B, END_FACT_C, END_TYPE
    from SDO_UNITS_OF_MEASURE
    where UPPER(UNIT_OF_MEAS_NAME) = UPPER(CONVERT_TO_UNIT);


    IF START_TYPE != END_TYPE THEN 
        RAISE_APPLICATION_ERROR(-20808, 'ERROR: Units ' || STARTING_UNIT || ' and ' || CONVERT_TO_UNIT || ' are not of same type.');
    END IF;

    METRES := VALUE * START_FACT_B / START_FACT_C;
    RETURN (METRES * END_FACT_C / END_FACT_B);
END;
